Positioning

定位不是“超级专家 Agent”，而是信息安全 AI 转型的总入口与协同枢纽

Aegis 的核心价值不在于替代 AISOC、AIGRC、AIAppSec 的专业判断，而在于把已有和持续建设中的能力组织成一个可统一接入、统一路由、统一治理、统一观察的系统。

02

它要做什么

统一接收来自人员和系统的请求

统一做意图识别、分发与上下文承接

统一汇总多 Agent 与多组件结果

统一提供治理、审计和管理视图

它不做什么

不替代专业域 Agent 的深度判断

不重构所有底层安全平台

不把 A2A Task 当作业务状态机

不在第一阶段追求高自治

为什么现在需要它

已有安全 AI 能力入口分散，难以统一使用

不同 Work Agent 缺少协同视图与可治理链路

领导层需要统一观察建设成效与组织价值

架构层需要稳定的扩展位点，而不是一次性大平台

Five-Layer Blueprint

五层架构把“入口、协同、执行、基础设施、治理输出”清晰分开

这让 Aegis 既能成为跨域枢纽，又不会把下层专业职责卷到自己身上。它是一个结构上“轻”、连接上“强”、治理上“实”的中枢。

03

统一入口层

回答“从哪里进入、谁统一反馈”的问题。所有人机请求与系统请求，先归集到 Aegis 统一受理。

协同控制层

回答“如何正确分发、如何可控协同”的问题。这里是 Aegis 作为中枢最关键的能力区。

Work Agent 层

回答“谁负责专业处理”的问题。专业域能力留在专业 Agent，不向 Aegis 上卷。

安全基础设施层

回答“实际动作由谁执行”的问题。IPS、WAF、EDR、PAM、AI 安全网关、AI IAM、XTIP、SOAR.AI 都在这里。

治理与度量输出层

回答“如何让领导和运营看得懂、管得住”的问题。统一输出价值度量，而不是只留技术日志。

Boundaries

边界清晰，是 Aegis 能长期扩展又不失控的前提

设计里的关键原则，是让 Aegis 做“统筹与连接”，而不是做“吞掉一切的超级平台”。只有边界先清楚，组织才敢持续接入更多 Agent 与更多安全组件。

04

Aegis 的边界

负责：入口 / 路由 / 编排 / 汇总 / 审计
  • 统一意图识别与任务分发
  • 统一上下文承接与状态管理
  • 统一治理和结果输出
  • 不替代专业域判断，不替代底层执行引擎

Work Agent 的边界

负责：专业分析 / 专业判断 / 专业动作建议
  • AISOC 处理运营与事件响应
  • AIGRC 处理控制、证据、合规内控
  • AIAppSec 处理研发安全与应用暴露
  • 不对外形成分散入口，不绕开治理要求

基础设施层的边界

负责：检测 / 控制 / 取证 / 自动化执行
  • 承载可调用的底层安全能力
  • 为专业 Agent 提供执行支撑
  • 不承担统一交互
  • 不承担跨域业务语义理解与流程编排

Collaboration

协同不是单一路径，而是“统一入口模式 + 预定义协同模式”的双模式体系

模式一解决“新请求怎么进入”；模式二解决“进入流程后的 Agent 如何高效联动”。两者不是互斥关系，而是前后接续、最终统一回流到 Aegis 视图与治理层。

05

模式一：统一入口模式

1
进入 Aegis

所有来自人员或系统的新请求，先进入 Aegis。

2
统一识别

执行意图识别、上下文组装、权限与策略检查。

3
统一分发

路由到 AISOC / AIGRC / AIAppSec，或拉起多 Agent 任务。

4
统一回收

输出统一回答、工单动作、审计记录和管理视图更新。

模式二：预定义协同模式

1
Agent 内触发

某个 Work Agent 在既定流程中触发预定义工作流。

2
直接协作

Agent 之间直接推送信息、获取信息、推送判断或触发下一步动作。

3
无重复识别

这条链路无需重新做意图识别，但依然受治理与状态跟踪约束。

4
统一回流

流程状态、结果、证据与度量最终同步回 Aegis。

Incident → Compliance

AISOC 识别事件后，触发 AIGRC 检查控制影响与证据义务，最后由 Aegis 汇总处置与治理影响。

App Risk → Ops

AIAppSec 发现高风险暴露后，拉通 AISOC 强化监测与响应准备，保持研发侧与运营侧同步。

AI Usage → Guardrails

入口请求在 Aegis 侧经过 AI 安全网关与 AI IAM 限制，再交给专业 Agent 继续处理。

Roadmap

建设路径强调“先可见、再协同、后深化自治”

路线设计遵循渐进式演进，而不是一次性重做所有安全系统。先让入口和治理站稳，再把协同闭环做厚，最后再扩大覆盖面和自动化深度。

06

第一阶段：近期

目标：建立 Aegis 统一入口和基础可见性。

  • 接入 AISOC / AIGRC / AIAppSec
  • 打通路由、权限、审计、结果汇总
  • 形成初步统一视图
产出：可用入口 + 基础治理

第二阶段：中期

目标：形成稳定的多 Agent 协同闭环。

  • 标准化预定义工作流
  • 增强底层安全组件联动
  • 证据共享与流程模板化
产出：稳定协同 + 可量化闭环

第三阶段：远期

目标：演进为企业级安全 AI 能力中枢。

  • 扩大覆盖域和 AI 原生安全服务
  • 基于策略与反馈持续优化协同
  • 在可控前提下提升自动化深度
产出：规模化能力网络 + 更强组织价值

Protocol Role

A2A 在 Aegis 里是“标准通信底座”，不是“业务协同本身”

A2A 解决的是 Agent 间如何发现、发送消息、跟踪任务、返回结果；真正的业务语义、业务状态、工作流治理和统一视图，仍然必须由 Aegis 及其业务扩展协议承担。

07

A2A 提供的三类基础能力

标准化消息交换：通过 Message / Part 统一文本、文件、结构化数据。

标准化任务跟踪：通过 Task / TaskStatus / TaskState 提供交互级生命周期。

标准化调用接口：通过 SendMessage / GetTask / CancelTask / Streaming 建立统一访问方式。

Limits

A2A 可以让消息“到得了、查得到、回得来”，但不能保证业务“理解一致、流程正确、治理完整”

这是附录中最需要被管理层和架构层同时理解的一点。A2A 不是业务协议，也不是安全治理架构；它只是一层必要但不充分的通信标准。

08
维度 A2A 能做的 A2A 不能直接做的 应由谁补齐
通信 Agent 间消息传输、任务查询、流式回传 消息背后的业务意义与责任划分 Aegis 协同控制层
状态 TaskState 描述协议交互状态 事件阶段、工单阶段、证据阶段、审批阶段 业务工作流与状态模型
语义 可传输结构化数据 告警、风险、控制、证据、策略等统一对象语义 业务对象模型
治理 可暴露标准化任务接口 权限、审批、审计、异常回退、人工接管 Aegis 治理与审计机制

为什么说 A2A Task 不是业务状态

`TASK_STATE_WORKING` 只能说明某个 Agent 正在处理请求，不能说明“事件正在处置中”； `TASK_STATE_COMPLETED` 只能说明协议调用完成，不能说明“业务闭环已结束”。 这意味着协议状态和业务状态必须在模型上显式分层。

A2A 之上必须补齐什么

业务对象模型、路由规则、协作规则、权限与身份、审计要求、异常机制、统一视图。 没有这些补充，A2A 只能做到互联互通，做不到真正的协同治理。

Dual-Channel Session Control

双通道会话跟随控制协议，把“通信”和“控制”从同一条会话里拆开

这不是另起炉灶替代 A2A，而是在 A2A 之上增加会话级控制层：谁在前台、输入路由给谁、用户跟随谁、何时回到主会话，全都被显式控制和可审计化。

09

通信通道

解决：消息怎么传、任务怎么查、结果怎么回
  • 基于 A2A 的 SendMessage / GetTask / CancelTask / Streaming
  • 负责 Agent 间标准消息与任务交互
  • 强调“调用”和“结果交换”

控制通道

解决：当前谁在说话、谁接收输入、何时切回主流程
  • 会话焦点切换
  • 前台跟随与输出观察
  • 输入绑定、返回主会话、退出与重置控制

会话分层

主会话负责总控与结果归集，子会话负责局部前台协作。

输入输出分离

输入路由和输出跟随被独立控制，不再绑死在一条单通道会话上。

焦点返回

子会话结束，不等于系统结束；可以自然回到主会话继续推进。

主会话保护

主会话忙碌时拒绝额外控制输入，避免状态混乱和待回放堆积。

Methods

标准方法解决通信，扩展动作解决控制

双通道协议的方法可以分成两组：左边是沿用 A2A 的标准方法，右边是围绕前台会话跟随和控制切换建立的扩展动作语义。

10

A2A 标准方法 通信底座

  • SendMessage：发起新请求或向既有任务追加消息
  • GetTask：查询任务状态、历史和产物
  • CancelTask：取消正在处理的任务
  • SendStreamingMessage：在支持时获取增量返回
  • AgentCard：识别远程 Agent 能力并完成发现

控制扩展动作 会话控制层

  • delegate_ext(is_loop=true)：创建并切入前台子会话
  • /main：结束当前子会话并返回主会话
  • /exit：在子会话内等价于返回主会话，在主会话内代表退出
  • /new：重置主会话上下文
  • 可抽象为 FocusSession / BindInput / FollowOutput / ReturnToMain
关键理解： A2A 官方标准并没有定义这些控制动作。 它们属于建立在 A2A 通信能力之上的业务扩展语义，用来补齐复杂协同里“谁在前台、谁接收输入、什么时候切回主流程”的控制需求。

Closing

真正让 Aegis 成立的，不是单个 Agent 或单个协议，而是“架构边界 + 协同机制 + 协议分层”三者同时清晰

总体蓝图回答“系统应如何组织”；A2A 回答“Agent 如何互联”；双通道协议回答“复杂协同如何被跟随和控制”。 三者叠在一起，Aegis 才真正具备走向组织级安全 AI 能力中枢的基础。

11

架构层结论

Aegis 应保持“轻中枢”定位：统一入口、统一路由、统一治理、统一输出，不吞并专业判断。

协议层结论

A2A 只是标准通信底座，业务状态、业务语义、业务工作流必须由 Aegis 业务层补齐。

协同层结论

双通道会话跟随控制协议，使复杂协作过程可前台观察、可切换、可接管、可统一回流。
